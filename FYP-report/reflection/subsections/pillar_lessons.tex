\section{Design Lessons by Pillar}
\label{sec:pillar-lessons}

This section covers the design lessons from building and testing MERIT. We will discuss the reasoning that was behind each of the 
choices we made to provide inspiration for future work.

\subsection{Pillar 1: Treat ingestion as lossy}
\label{sec:pillar1-reflection}

As we saw in Study~04, parser and dictionary limits mean extensible metrics are often configured on incomplete CV and JD extractions.
LinkedIn can help as a fallback for missed CV experience, while GitHub can help as a fallback for missed CV skills. These sources should
be treated as part of the ingestion strategy, not as optional extras.

\subsection{Pillar 2: Missing data $\neq$ low skill}
\label{sec:pillar2-reflection}

The lesson here is that the current approach to our scoring policy treats absent GitHub activity as weak, or no evidence, rather than
as missing data. A missing data source, such as GitHub, should not automatically be read as low competence, especially when a candidate's
most representative work is private. Future work should cover appropriate approaches to handle missing data.

\subsection{Pillar 3: Explain conflict, not only attribution}
\label{sec:pillar3-reflection}

Glass-box output is central to MERIT's design, from Shapley values to Blind Mode. Negative source contributions and integrity-driven reversals
show the interface working as intended. However, Study~10 shows that the system can still produce a poor rank unless conflict and uncertainty
are surfaced clearly enough for recruiters to override the engine. Clearer communication of both conflict and uncertainty is a
promising direction for future work.
