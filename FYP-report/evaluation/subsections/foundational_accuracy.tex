\section{Foundational Ingestion \& Evaluating Pillar 1: Extensibility}
\label{sec:foundational-ingestion-extensibility}

In real-world applications, the performance of MERIT is heavily dependent on the accuracy of its information
retrieval (IR) layer, also referred to as the ingestion and extraction pipeline. This layer is responsible
for parsing structured and unstructured candidate documents and job descriptions. For this evaluation, we
focus primarily on unstructured data formats (PDF-based CVs and JDs), as structured profile feeds
(such as raw LinkedIn or GitHub JSON payloads) present trivial extraction tasks that do not require
layout-based parsing.

Our evaluation consists of two separate audits measuring the accuracy of information retrieval for CVs
and JDs, respectively. Both studies compare the system's extraction outputs against human-verified
ground-truth datasets, evaluating a total of 30 CVs (balanced across diverse layouts) and 15 JDs.
Per-document score distributions are in Appendix~\ref{appendix:ingestion-charts}, 
summary tables remain below.

\subsection{Ingestion Architecture \& Methodological Mechanics}
The ingestion layer implements custom parsing scripts located in \texttt{/backend/core/parsers}. These
scripts ingest, sanitise, and extract key entities from unstructured PDF files through the following
pipeline:

\subsubsection{Text Extraction and Sanitisation}
For both CVs and JDs, raw text and embedded hyperlinks are extracted using the Python libraries
\texttt{pdfplumber} and \texttt{pdfminer} \cite{pdfminer}. The extracted raw text is then sanitised by
removing null bytes (\texttt{\textbackslash u0000}) and redundant whitespace characters.

\subsubsection{Named Entity Recognition (NER) \& Contact Parsing}
To extract candidate identities, the first 500 characters of the document are processed by spaCy's
English NER pipeline (\texttt{en\_core\_web\_sm}) \cite{spacy2020}. Candidate names are identified by
matching the \texttt{PERSON} entity label. Email addresses are extracted using targeted regular expressions,
while phone numbers are validated and normalised using Google's \texttt{phonenumbers} library. Professional
links are matched against LinkedIn and GitHub domains, falling back to heuristic username matching patterns
where explicit URLs are absent.

\subsubsection{Dictionary-Based Skill Extraction}
Skills are identified using a dictionary-based matching engine that references a local index file,
\textit{skills\_data.json}. This file is dynamically updated via an automated ingestion script that scrapes
Devicon, a comprehensive community database of developer technology names and logos. Case-insensitive matches
are performed using regular expressions. To prevent false positive matches, strict word boundaries are enforced
using the \texttt{\textbackslash b} regex metacharacter. This ensures, for example, that the programming language
"R" is not falsely matched within the text token "Ruby".

\subsubsection{Heuristic Section Segmentation}
When parsing CVs, the extraction engine segments the document text by identifying commonly used section
headings (e.g., 'Experience', 'Education', 'Projects'). Bullet points and text blocks are split along common
delimits (pipes, colons, and commas) to extract roles, companies, and institutions, while regular expressions
extract and normalise temporal date ranges.

\subsection{Mathematical Performance Metrics}
\label{subsec:mathematical-metrics}

For our evaluation, we will be evaluating both set-based and string-based comparisons between the ground truth,
and the extracted data for our three core categories (skills, experience and identity).

\subsubsection{Skill Set Metrics}
Given $S_E$ as the set of extracted skills and $S_{GT}$ as the set of ground truth skills, we can
formulate the precision ($P_{skills}$), recall ($R_{skills}$), and F1-score ($F1_{skills}$) as follows:
\begin{equation}
    P_{skills} = \frac{|S_E \cap S_{GT}|}{|S_E|}
\end{equation}
\begin{equation}
    R_{skills} = \frac{|S_E \cap S_{GT}|}{|S_{GT}|}
\end{equation}
\begin{equation}
    F1_{skills} = \frac{2 \cdot P_{skills} \cdot R_{skills}}{P_{skills} + R_{skills}}
\end{equation}

\subsubsection{Experience Recall}
For experience extraction, we mainly care about recall, since if we miss an experience, we might miss out on a
correct match. False positives are not a significant concern for this metric due to the strict nature of experience
extraction.

Given a set of ground truth experiences $E_{GT}$, and a set of extracted experiences $E_E$, we define a
successful match as follows:
\begin{equation}
    \text{Match}(e_{gt}) = \mathbb{I}\left(\exists e_e \in E_E \text{ s.t. } \text{norm}(e_{gt}) \subseteq \text{norm}(e_e)\right)
\end{equation}

Experience Recall ($R_{exp}$) is thus:
\begin{equation}
    R_{exp} = \frac{1}{|E_{GT}|} \sum_{e_{gt} \in E_{GT}} \text{Match}(e_{gt})
\end{equation}

\subsubsection{Identity and Metadata Matching}
For the identity matches of both candidate CVs and JDs, we care about exact matches, so we do not compute
precision and recall, but rather a match rate. For candidate CVs, we evaluate name and email matches as
exact binary matches, we do the same for job titles in JDs. However, for the company name of the JD, we
perform a partial containment indicator, checking to see if the ground truth label is a substring of the
extracted label.

\subsubsection{Qualitative Analysis \& Parsing Volatility}
This audit reveals a major structural insight that explains the average performance metrics and the
volatility observed in the per-document distribution (Figure~\ref{fig:cv_parser_accuracy}).

The structural layout of the CV will significantly affect the performance of the parser, a layout
vulnerability that has been recently formalised in large-scale multi-template resume parsing benchmarks
such as \textit{ResumeBench} \cite{emnlp2025resumebench}. It is critical to note the difference of
83.33\% in Experience Recall between single-column and multi-column CVs for this parser specifically.
This occurs because underlying PDF text extractors (such as \texttt{pdfminer} \cite{pdfminer}) extract
characters based on spatial coordinates, merging horizontally adjacent text blocks into single, interleaved
text strings. By combining these horizontal columns, the parser's line-by-line section segmentation
heuristics fail completely, generating severe false negatives. This structural fragility of parsing
unstructured PDF data provides a strong, primary technical justification for integrating structured,
schema-validated profile sources (such as a standardised LinkedIn profile) to deterministically bypass
coordinates extraction errors.

In real-world recruitment applications, commercial Applicant Tracking Systems (ATS) suffer from the
same structural parsing failures due to their reliance on standard spatial PDF-parsing libraries. It is
for this reason that job candidates are universally advised in popular employment guides to use simple,
single-column linear CV templates \cite{marr2019, kelly2021}. This formatting constraint introduces a
profound, systemic bias: automated screening engines inadvertently filter out highly qualified candidates
simply because their visually rich or modern multi-column CV designs confuse the parser. As highlighted
by algorithmic hiring policy reviews \cite{raghavan2020mitigating}, this selection pressure does not
represent a fair or accurate representation of candidate capability, but rather a technical bias towards
candidates with simple, linear templates—favoring "ATS-friendly" formatting over genuine professional merit.

\subsubsection{Qualitative Analysis \& Strategic Ingestion Failure Cases}




Unlike CVs, which serve as concise professional summaries, Job Descriptions (JDs) are inherently
marketing-oriented documents designed to attract candidates. As a result, they are often dense with corporate
boilerplate, such as institutional missions, values, wellness benefits, and equal opportunity clauses.
This structural noise significantly degrades parsing accuracy. Because section headings are frequently
styled with generic tags (e.g., 'About Us' or 'Who We Are'), the parser struggles to isolate core technical
qualifications from general corporate copy. This lexical contamination is directly reflected in the metrics,
driving a low Skill Recall of 34.13\% and restricting Skill Precision to 60.02\%, as the extraction engine
fails to separate critical technology requirements from marketing jargon.

The most striking result in the JD audit is the company name match rate, which collapses to a mere
6.67\% (successful on only 1 out of 15 documents). A deep-dive error analysis reveals that this failure
is primarily structural rather than semantic: employers rarely represent company names in machine-readable
plain text within document headers. Instead, corporate identities are typically embedded as images.
Because text extractors like \texttt{pdfminer} cannot extract alphanumeric tokens from embedded images,
the parser is forced to rely on the unstructured "About Us" section. Consequently, if the employer's
name is omitted from this section, or referred to via generic pronouns (e.g., "we", "our company"), the
metadata extraction fails entirely. In the context of the broader MERIT architecture, however, this
extraction bottleneck is non-critical, as identifying the employer name from the JD is a convenience
feature rather than a core functional requirement. This metric will have no impact on the overall
ranking of candidates.



\subsection{Study 13: Metric Prediction and Recruiter Alignment}
\label{subsec:study-13-metric-prediction}

While Studies~04 and~05 test skill \emph{extraction}, Study~13 evaluates whether TF-IDF \emph{priority} suggestions align with recruiters.
We test \textbf{H1} (positive Spearman $\rho$ within each role) and \textbf{H2} (low MAE versus consensus on MERIT's 1--5 scale,
where 1 = essential and 5 = optional).

\subsubsection{Method}
For the generation of the market corpus, we leverage Tonic.ai to generate $N = 100$ synthetic JDs. This provides a relatively realistic
dataset to aid the computation of IDF. Three hold-outs are excluded from this corpus (Junior Frontend/React, Senior ML, Lead DevOps/Platform)
so that IDF is not biased toward the skills under test. We use one pooled document-frequency table, in other words, we do not partition the corpus by
job title or role family.

Our study includes four evaluators (two recruiters and two academic peers) rating study-controlled skill lists under a blind protocol
(JD and metrics are shown, but TF-IDF suggestions are hidden). Consensus is the median of the four integer ratings. We compare this to
TF-IDF suggestions using exact match, MAE, and Spearman $\rho$. Charts are in Appendix~\ref{appendix:metric-prediction-charts}.

Table~\ref{tab:study-13-metric-alignment} shows that TF-IDF is not well calibrated as an 
automatic 1--5 default. Alignment is role-dependent: junior frontend performs best, senior ML worst, and lead DevOps sits between the two. 
Aggregate $\rho = 0.25$ is positive but not significant at $\alpha = 0.05$.

Of the 27 skill--role pairs, eight achieve exact agreement and a further nine differ by only one Likert band (63\% within $\pm 1$).
These near-misses show that the engine can be directionally useful, but it is not safe as a silent default. The largest gaps include
Python on the senior ML hold-out (consensus P-1, TF-IDF P-5): Python appears in many role templates in the pooled corpus, so global IDF
treats it as common for that vacancy even though recruiters rated it essential. Similar headline failures appear for the PhD/MSc
education line and CUDA on the same hold-out.

\subsubsection{Limitations and implications}
\label{subsubsec:study-13-limitations}

This is a pilot study ($N = 4$ evaluators, $n = 27$ pairs), and the results are not generalisable to the wider market.
Due to the constraints of the project, we could not run a larger panel or ingest a longitudinal corpus of real JDs within scope.

Another limitation is cross-role IDF interference. Take Python as an example: it is common across many domains in our pooled corpus,
so IDF can down-rank it on a senior ML hold-out even when recruiters treat it as essential. The same pooling effect can mis-rank
role-specific requirements in niche hiring (e.g.\ quantitative development, where Python may be a differentiator rather than baseline
noise) because document frequency is shared across unrelated templates.

Participants were also not asked to spread ratings evenly across P-1--P-5, whereas TF-IDF applies per-JD min--max normalisation and
tends to stretch suggestions across the band. Recruiters instead polarised toward P-1/P-2 for headline skills and P-4/P-5 for hygiene
requirements, with little use of P-3. That mismatch can depress exact-match rate and $\rho$ without invalidating the pilot, it is a
comparison caveat we report alongside the within-$\pm 1$ band counts above. Cross-study validity threats are summarised in
Section~\ref{sec:threats-to-validity}.

To conclude, we believe \textbf{H1} is only partly supported and \textbf{H2} is not met. The pilot supports treating TF-IDF as
suggestions with mandatory human overrides, rather than silent defaults. Further work could stratify IDF by role family and study how
recruiters distribute priorities without forcing an even spread across the Likert scale.


\subsection{Downstream Mitigation and Architectural Rationale}
\label{subsec:downstream-mitigation}

These quantitative findings empirically justify two of the three core architectural pillars of the MERIT system.

\subsubsection{Multi-Source Fusion (Pillar 2)}
As shown from the CV parser's evaluation, unstructured data extraction is often noisy, leading to
incomplete skill extraction. Through providing structured data sources (such as LinkedIn and GitHub data),
high-skill candidates can still be successfully considered within MERIT, even if their CVs are poorly formatted.

\subsubsection{Glass-Box Explainability (Pillar 3)}
The high volatility of heuristic parsing (evident in Appendix~\ref{appendix:ingestion-charts}) underscores the risk of relying on automated screening platforms
that operate as black boxes. By displaying detailed, interactive candidate reports that show exactly
where every matching skill was extracted and how it was verified, MERIT equips recruiters to immediately
audit, correct, and override parsing discrepancies. This transparency ensures highly valid hiring
decisions despite the persistent limitations of automated document parsing.
